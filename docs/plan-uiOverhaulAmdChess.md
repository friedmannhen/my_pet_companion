# Plan: Nested-Drawer UI Overhaul + Chess + Forfeit + Leaderboard Filters

Big scope across 4 phases, ordered per your call (UI restructure → small wins → Chess last). No new physics/rendering library — framer-motion springs already in the codebase are enough (drawer spring `stiffness:300/damping:32`, radial-menu stagger `stiffness:320/damping:22`); this whole plan is architecture-first since art assets aren't ready yet.

**Phase 0 — Nested "closet of drawers" UI restructure**
1. Split [SideDock.tsx](apps/desktop/src/game/SideDock.tsx)'s single `view`-switched drawer into independent floating panels: the main drawer keeps only the ribbon nav + stat progress bars (today's "home" minus Kitchen&toybox). Kitchen becomes its own always-independently-toggleable drawer (promoted out, stays open during drag-throw so the pet stays reachable). Online/Friends/Ranks/Quests/Awards/History/Settings become their own secondary sliding panels, mutually exclusive (one "info" panel open at a time), stacked beside the main drawer — matches your "combination" answer.
2. Each new panel needs its own `data-interactive` root + `pointerEvents:"auto"` subtree per the pattern in [overlay/useHitTest.ts](apps/desktop/src/overlay/useHitTest.ts) / [overlay/clickableOverride.ts](apps/desktop/src/overlay/clickableOverride.ts) — this is the part most likely to regress click-through if done carelessly (there's a documented single-resync-after-override bug already fixed once; don't reintroduce it with more panels).
3. Extend [useRibbonPrefs.ts](apps/desktop/src/game/useRibbonPrefs.ts) with new fields for which panels are open/positioned (kitchen open bool, active info-panel id) so layout survives restarts.
4. **New mechanic (your idea, folded in here since it's the same drawer work): poop cleanup.** Poop items spawn probabilistically after feed events (new hook in [decay.ts](packages/pet-core/src/decay.ts)/[rules.ts](packages/pet-core/src/rules.ts) style). Spawn sequence (your refinement): the pet does a quick wiggle animation first (reusing the same transient-trigger-class pattern already used for `petSeatedWiggling`/`egg-anim-wiggle` — one new CSS keyframe, no new art needed), then the poop item smoothly slides/fades in just below the pet (a framer-motion `initial`→`animate` on y+opacity, not an instant pop-in) before it's drag-dropped onto a trash can living inside the Kitchen drawer, reusing the exact grab/drag-handoff pattern Kitchen already has for food/ball. Cleaning a poop grants a small care-points + happiness bump (smaller than a full feed/wash/pet action — exact numbers are a tunable placeholder, not final). Track a persistent `poopCleanedCount` on the pet's save/row, same place as the existing feed/wash/pet/throw-ball counters that already feed the leaderboard's "interactions" metric, so a future `poopClean25/100/250`-style achievement tier can be added later mirroring the existing feed/wash/pet/play tiers in [achievements.ts](packages/pet-core/src/achievements.ts) — just the counter plumbing now, no achievement definition yet. Because this adds a new small trickle of care points, re-check the combined effect against the evolution-pacing baseline in [rules.ts](packages/pet-core/src/rules.ts) (the "regular player" calibration: hatch ~day 5, adult ~day 16, final ~day 45) once real spawn-frequency/reward numbers are picked, so the extra points don't quietly speed up progression more than intended.
5. *Depends on 1-4.* Reskin pass deferred — this phase intentionally keeps current placeholder visuals; book-style (image 1) vs wood/vine (image 2/3) art is a separate follow-up once assets exist.
6. **New tweak (your addition): room presence count display.** Replace the current per-player repeated green-dot badge (`"🟢".repeat(...)` in [SideDock.tsx](apps/desktop/src/game/SideDock.tsx) ~L156-166, fed by `useGroupPresenceCount`) with a single `{count} 🟢` numeric + online-icon format in the same spot — rendering change only, no new data plumbing needed.
7. **Idle liveliness — framer-motion breathing + random gestures** (your addition; off-topic from the drawer work itself, but same no-new-art spirit so it fits this phase). Today's idle breathe is a plain CSS keyframe (`pet-anim-idle-breathe` in [petAnimations.css](apps/desktop/src/game/petAnimations.css), gated by the `idleBreathing` boolean in [GameView.tsx](apps/desktop/src/game/GameView.tsx) ~L1580-1591: pet alive, hatched, not sleeping/moving/busy, no other pulse active). Proposal: drive the breathing squash itself with framer-motion instead — a `scaleY` loop via `animate()`/`useAnimationControls` with `repeatType:"mirror"` (same technique already used in [throwPhysics.ts](apps/desktop/src/game/throwPhysics.ts)/[curlPhysics.ts](apps/desktop/src/game/curlPhysics.ts)) for smoother, interruptible easing than a fixed-percentage CSS keyframe, using the exact same gating condition — just a different animation engine, no new triggers. Layer on top a randomized "liveliness" scheduler: while `idleBreathing` is true, fire one small one-off gesture every ~6-20s at random (ear/tail twitch, a quick blink via the existing wink/blink asset swap, a brief look-around tilt) — reusing the same transient-trigger-class pattern already used for `petHappyWiggling`/`petSeatedWiggling`, just sourced from a random timer instead of a game event. No new art required for v1 (blink/wink assets already exist); richer gestures wait for future pet art.

**Phase 1 — Universal forfeit/give-up button** *(parallel with Phase 2, both small)*
8. Add a `forfeit` broadcast kind in [useRoom.ts](apps/desktop/src/online/useRoom.ts) (alongside existing `challenge`/`invite`/`decline` patterns ~L786-845), with handlers that resolve the match as a loss for the quitter + win for the opponent, and call `record_minigame_result` accordingly.
9. Wire a "Give up" button into [RockPaperScissors.tsx](apps/desktop/src/game/minigames/RockPaperScissors.tsx), [TargetToss.tsx](apps/desktop/src/game/minigames/TargetToss.tsx), and the new Chess UI (Phase 3). Not added to Battle (resolves instantly already, per your call).

**Phase 2 — Per-minigame leaderboard filter** *(parallel with Phase 1)*
10. Extend [useLeaderboard.ts](apps/desktop/src/game/useLeaderboard.ts) (currently only reads `pets` table stats, never touches `minigame_scores`) to optionally query `minigame_scores` filtered by `game_code`, with a per-game sort key (Target Toss = ascending `best_score` distance; RPS/Chess = descending `games_won`).
11. Add a game-picker selector in the "ranks" panel, same pill-button style as the existing carePoints/interactions/evolutionStage metric pills.

**Phase 3 — Chess minigame** *(depends on Phase 0's panel system + Phase 1's forfeit hook; biggest phase, last)*
12. New migration `chess_games` table: `group_id`, `player_a_id`, `player_b_id`, `board_fen`, `move_history jsonb` (full SAN log — needed for resume; the in-panel "recent moves" widget in item 17 just reads a slice of this same array, no separate persistence), `current_turn`, `status ('active'|'finished'|'abandoned')`, `winner_id`, `result_reason`. **Untimed only for v1** (your revision — no `mode`/time-bank/clock columns at all): the game simply lives in that room until checkmate, resignation, or a player explicitly ends it. RLS: only the two players can write; anyone who `shares_group_with(group_id)` can SELECT (spectator read access) — mirrors the existing `minigame_scores` RLS shape.
13. Add `chess.js` (MIT) as a dependency for legality/check/checkmate/castling/en passant. Client validates moves locally before writing; RLS enforces `current_turn = auth.uid()` on update (turn-ownership only, not full legality) — flagged explicitly as an MVP trust tradeoff, same class of decision as the existing "Realtime private-channel authorization is a hardening follow-up" note in `useRoom.ts`.
14. **King piece = the player's own live pet asset** (your addition): each king square renders that player's actual pet sprite at its current evolution stage — reusing the same idle/blink assets and CSS keyframes (`idle-breathe`, wink swap) already used in [GameView.tsx](apps/desktop/src/game/GameView.tsx) — instead of a generic king glyph. The opponent's king shows their pet the same way, reusing [RemotePets.tsx](apps/desktop/src/online/RemotePets.tsx)'s existing "fetch and render another player's pet sprite" logic rather than re-deriving it.
15. **Per-player board orientation** (your addition): each viewer always sees their own side at the bottom — a pure render-side flip (reversed rank order + mirrored file/rank labels) driven by `myColor`; no separate data representation needed. Spectators default to White-at-bottom.
16. Board shows standard algebraic file/rank labels (a-h / 1-8, correctly mirrored per item 15) so moves read as normal chess notation.
17. **In-panel "recent moves" widget** (your addition): last 3 moves per player, read directly off the already-loaded `move_history` array/session reducer state — a UI slice only, no DB table, query, or extra persistence beyond what item 12 already stores.
18. **Live-game visibility** (your addition, several related pieces):
    - Room card badge (same spot as item 6's presence-count tweak) shows when a room has an active chess game, styled differently when the viewer is one of the two players vs. just aware one exists.
    - A persistent badge on the main ribbon/collapsed tab (same affordance as SideDock's existing "claimable rewards" badge on the closed tab) shows the player has an ongoing game, visible even while they're elsewhere in the app or out of that room.
    - **Minimize control**: the Chess panel collapses to a small floating chip instead of closing — the game/subscription stays alive in the background so it doesn't block other desktop work; reopening restores the board. Minimizing never forfeits.
    - Leaving the ROOM (disconnecting from the realtime channel) only pauses your visible presence — no forfeit, no clock to worry about since it's untimed; the game just waits for you. Leaving the GROUP (removing membership) is bigger — since the game is stored against `group_id`, warn the player first if it would strand an active game.
19. **"Poke" button** (your addition) inside the Chess panel: sends a targeted notification to the opponent via the existing [useNotifications.ts](apps/desktop/src/online/useNotifications.ts) user-inbox system (new `NotificationKind: "chess_poke"`), surfacing as the same toast-on-top-of-the-pet UI already used for friend requests/room invites ([GameView.tsx](apps/desktop/src/game/GameView.tsx) ~L2116-2147) — reaches them even if they've minimized the board or left the room, as long as their app is open (same accepted "missed if fully offline" tradeoff the notification system already documents).
20. New `apps/desktop/src/game/minigames/Chess.tsx` bringing 14-19 together: CSS-grid board (no new art beyond the pet-as-king reuse), legal-move highlighting via chess.js, resign/give-up button (Phase 1's hook), read-only spectator mode for non-participants, resumes from DB on room (re)join.
21. Challenge flow extends the existing `challenge()`/`inviteMinigame()` targeted-broadcast pattern in `useRoom.ts` (~L762-845) — no mode picker needed now that it's untimed-only.
