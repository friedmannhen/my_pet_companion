# Plan: Nested-Drawer UI Overhaul + Chess + Forfeit + Leaderboard Filters

Big scope across 4 phases, ordered per your call (UI restructure → small wins → Chess last). No new physics/rendering library — framer-motion springs already in the codebase are enough (drawer spring `stiffness:300/damping:32`, radial-menu stagger `stiffness:320/damping:22`); this whole plan is architecture-first since art assets aren't ready yet.

**Phase 0 — Nested "closet of drawers" UI restructure**
1. Split [SideDock.tsx](apps/desktop/src/game/SideDock.tsx)'s single `view`-switched drawer into independent floating panels: the main drawer keeps only the ribbon nav + stat progress bars (today's "home" minus Kitchen&toybox). Kitchen becomes its own always-independently-toggleable drawer (promoted out, stays open during drag-throw so the pet stays reachable). Online/Friends/Ranks/Quests/Awards/History/Settings become their own secondary sliding panels, mutually exclusive (one "info" panel open at a time), stacked beside the main drawer — matches your "combination" answer.
2. Each new panel needs its own `data-interactive` root + `pointerEvents:"auto"` subtree per the pattern in [overlay/useHitTest.ts](apps/desktop/src/overlay/useHitTest.ts) / [overlay/clickableOverride.ts](apps/desktop/src/overlay/clickableOverride.ts) — this is the part most likely to regress click-through if done carelessly (there's a documented single-resync-after-override bug already fixed once; don't reintroduce it with more panels).
3. Extend [useRibbonPrefs.ts](apps/desktop/src/game/useRibbonPrefs.ts) with new fields for which panels are open/positioned (kitchen open bool, active info-panel id) so layout survives restarts.
4. **New mechanic (your idea, folded in here since it's the same drawer work): poop cleanup.** Only possible once the pet has hatched (any evolution stage 1-3) — an egg (stage 0) never poops, so the spawn hook must gate on `!game.isEgg` the same way other post-hatch-only mechanics do. Poop items spawn probabilistically after feed events (new hook in [decay.ts](packages/pet-core/src/decay.ts)/[rules.ts](packages/pet-core/src/rules.ts) style). Spawn sequence (your refinement): the pet does a quick wiggle animation first (reusing the same transient-trigger-class pattern already used for `petSeatedWiggling`/`egg-anim-wiggle` — one new CSS keyframe, no new art needed), then the poop item smoothly slides/fades in just below the pet (a framer-motion `initial`→`animate` on y+opacity, not an instant pop-in) before it's drag-dropped onto a trash can living inside the Kitchen drawer, reusing the exact grab/drag-handoff pattern Kitchen already has for food/ball. Cleaning a poop grants a small care-points + happiness bump (smaller than a full feed/wash/pet action — exact numbers are a tunable placeholder, not final).
   - **Persistence (revised — must sync to the cloud, not just local):** since future achievements/statistics will key off this counter, `poopCleanedCount` needs the full plumbing the existing feed/wash/pet/throw-ball counters have, not just a local field: a new `poop_cleaned_count integer not null default 0` column on the `pets` table (new migration, granted the same as the other count columns), a matching `poopCleanedCount` field in [types.ts](packages/pet-core/src/types.ts)'s `PetSaveData` + default in [save.ts](packages/pet-core/src/save.ts), and both directions of the row↔save mapping in [petRow.ts](apps/desktop/src/supabase/petRow.ts) (mirrors `feed_count`/`feedCount` exactly). Once that's in place it rides the same upsert [usePetGame.ts](apps/desktop/src/game/usePetGame.ts) already does for every other counter — no separate sync path needed. This also means a future `poopClean25/100/250`-style achievement tier can be added later mirroring the existing feed/wash/pet/play tiers in [achievements.ts](packages/pet-core/src/achievements.ts) — just the counter plumbing now, no achievement definition yet.
   - **Pacing check (corrected):** [rules.ts](packages/pet-core/src/rules.ts) confirms the old daily normal-point cap was already deprecated in favor of the larger evolution-stage thresholds (`DEFAULT_STAGE_DAILY_NORMAL_POINT_CAP` is just `MAX_SAFE_INTEGER` now, kept only for save-schema compatibility) — there is no daily cap to reconcile poop points against. The only thing to re-check once real numbers are picked is that poop's small added trickle doesn't quietly pull the hatch ~day 5 / adult ~day 16 / final ~day 45 baseline earlier than intended, same as any other new point source.
5. *Depends on 1-4.* Reskin pass deferred — this phase intentionally keeps current placeholder visuals; book-style (image 1) vs wood/vine (image 2/3) art is a separate follow-up once assets exist.
6. **New tweak (your addition): room presence count display.** Replace the current per-player repeated green-dot badge (`"🟢".repeat(...)` in [SideDock.tsx](apps/desktop/src/game/SideDock.tsx) ~L156-166, fed by `useGroupPresenceCount`) with a single `{count} 🟢` numeric + online-icon format in the same spot — rendering change only, no new data plumbing needed.
7. **Idle liveliness — framer-motion breathing + random gestures** (your addition; off-topic from the drawer work itself, but same no-new-art spirit so it fits this phase). Today's idle breathe is a plain CSS keyframe (`pet-anim-idle-breathe` in [petAnimations.css](apps/desktop/src/game/petAnimations.css), gated by the `idleBreathing` boolean in [GameView.tsx](apps/desktop/src/game/GameView.tsx) ~L1580-1591: pet alive, hatched, not sleeping/moving/busy, no other pulse active). **Decision (picking for best future scaling): move to framer-motion**, not just for breathing but as the single animation authority for all future idle gestures too — drive the breathing squash with a `scaleY` loop via `animate()`/`useAnimationControls` with `repeatType:"mirror"` (same technique already used in [throwPhysics.ts](apps/desktop/src/game/throwPhysics.ts)/[curlPhysics.ts](apps/desktop/src/game/curlPhysics.ts)), using the exact same gating condition. **Required guard rail (closes a real integration risk):** all the other pet states (`pet-anim-walk`, `pet-anim-eat`, `pet-anim-happy`, etc.) are still plain CSS classes that set `transform` via keyframes on the same element; a framer-motion-driven inline `transform`/`scaleY` style persists on that DOM node and will silently keep overriding those CSS classes once `idleBreathing` turns false unless explicitly cleared. So the moment the `idleBreathing` gate flips off, the code must stop the animation controls AND reset the motion value back to its rest state (`scaleY.set(1)`, or unset the inline style entirely) before handing the element back to `bodyClass`. Layer on top a randomized "liveliness" scheduler, also framer-motion-driven for the same reason (interruptible one-offs compose more easily than stacking more CSS keyframe classes as gestures grow richer later): while `idleBreathing` is true, fire one small one-off gesture every ~6-20s at random (ear/tail twitch, a quick blink via the existing wink/blink asset swap, a brief look-around tilt), each gesture also resetting its own motion value on completion. No new art required for v1 (blink/wink assets already exist); richer gestures wait for future pet art.

**Phase 1 — Universal forfeit/give-up button** *(parallel with Phase 2, both small)*
8. Add a `forfeit` broadcast kind in [useRoom.ts](apps/desktop/src/online/useRoom.ts) (alongside existing `challenge`/`invite`/`decline` patterns ~L786-845), with handlers that resolve the match as a loss for the quitter + win for the opponent, and call `record_minigame_result` accordingly. This "Give up" path always produces a decisive result (one player's loss, the other's win) — it's the only forfeit route for RPS/Target Toss, which resolve in seconds and don't need anything more.
9. Wire a "Give up" button into [RockPaperScissors.tsx](apps/desktop/src/game/minigames/RockPaperScissors.tsx), [TargetToss.tsx](apps/desktop/src/game/minigames/TargetToss.tsx), and the new Chess UI (Phase 3). Not added to Battle (resolves instantly already, per your call).
10. **Chess-only addition (your revision — untimed games can sit unresolved indefinitely, so plain forfeit-on-quit isn't enough): a "Cancel game" flow, separate from "Give up".** Rationale: a decisive Give-up always scores a win/loss, but nobody should be able to just close the app to dodge that — so closing the app/leaving must NEVER auto-score a win for the remaining player. Instead:
   - **Mutual cancel:** either player can propose canceling; this sends a targeted request to the opponent (same transient-broadcast pattern as `sendTo` in [useNotifications.ts](apps/desktop/src/online/useNotifications.ts)). If the opponent explicitly accepts, the game ends with `status = 'abandoned'`, `result_reason = 'mutual_cancel'`, and **no `record_minigame_result` call for either side** — it simply frees up the pair to start a new game (see item 12's one-active-game-per-pair rule).
   - **Unreachable-opponent fallback:** mutual consent is impossible if the opponent never comes back, so a cancel proposal that goes unanswered while the target has shown no presence in the room for a grace period (placeholder: 48h, tunable) can be resolved unilaterally by the requester as `result_reason = 'opponent_unreachable'` — still no score impact for either side, just an escape hatch so a vanished opponent can't strand the game forever. This is deliberately NOT a short timer — untimed means no urgency for players who are both still around; it only kicks in for genuine abandonment.

**Phase 2 — Per-minigame leaderboard filter** *(parallel with Phase 1)*

11. Add a game-picker selector in the "ranks" panel, same pill-button style as the existing carePoints/interactions/evolutionStage metric pills.

**Phase 3 — Chess minigame** *(depends on Phase 0's panel system + Phase 1's forfeit hook; biggest phase, last)*
12. New migration `chess_games` table: `group_id`, `player_a_id`, `player_b_id`, `board_fen`, `move_history jsonb` (full SAN log — needed for resume; the in-panel "recent moves" widget in item 17 just reads a slice of this same array, no separate persistence), `current_turn`, `status ('active'|'finished'|'abandoned')`, `winner_id`, `result_reason` (now covers `'checkmate'|'resignation'|'mutual_cancel'|'opponent_unreachable'`, per item 10's expanded cancel/abandon flow — `'finished'` only for `checkmate`/`resignation`, `'abandoned'` for the two no-score-impact endings). **Untimed only for v1** (your revision — no `mode`/time-bank/clock columns at all): the game simply lives in that room until checkmate, resignation, or a mutual/unreachable cancel. **RLS fix:** spectator SELECT must use `is_group_member(group_id)`, NOT `shares_group_with(...)` — `shares_group_with(other uuid)` compares against a *user* id ("does `auth.uid()` share a group with this user"), so passing a `group_id` into it would silently never match; `is_group_member(gid uuid)` is the existing helper that actually checks group membership by group id (mirrors the `minigame_scores` RLS shape, just with the correct helper). Writes stay restricted to the two players. **One active game per pair (your call):** a partial unique index on `(least(player_a_id, player_b_id), greatest(player_a_id, player_b_id))` where `status = 'active'` blocks a new challenge between the same two players until the existing one resolves (finished OR abandoned) — the challenge flow (item 21) should check-and-surface this as a friendly "you already have a game with them" message rather than a raw constraint error. **Multiple concurrent games per room (your call):** no cap on simultaneous games between *different* pairs in the same room — see item 18's picker.
13. Add `chess.js` (**BSD-2-Clause**, not MIT — corrected) as a dependency for legality/check/checkmate/castling/en passant. Client validates moves locally before writing; RLS enforces `current_turn = auth.uid()` on update (turn-ownership only, not full legality) — flagged explicitly as an MVP trust tradeoff, same class of decision as the existing "Realtime private-channel authorization is a hardening follow-up" note in `useRoom.ts`.
14. **King piece = the player's own live pet asset** (your addition): each king square renders that player's actual pet sprite at its current evolution stage — reusing the same idle/blink assets and CSS keyframes (`idle-breathe`, wink swap) already used in [GameView.tsx](apps/desktop/src/game/GameView.tsx) — instead of a generic king glyph. The opponent's king shows their pet the same way, reusing [RemotePets.tsx](apps/desktop/src/online/RemotePets.tsx)'s existing "fetch and render another player's pet sprite" logic rather than re-deriving it. **Edge case (your call): pet death mid-game is not handled** — accepted as a non-concern for v1, no forfeit/placeholder logic needed if a king's owning pet dies while a game is active.
15. **Per-player board orientation** (your addition): each viewer always sees their own side at the bottom — a pure render-side flip (reversed rank order + mirrored file/rank labels) driven by `myColor`; no separate data representation needed. Spectators default to White-at-bottom.
16. Board shows standard algebraic file/rank labels (a-h / 1-8, correctly mirrored per item 15) so moves read as normal chess notation.
17. **In-panel "recent moves" widget** (your addition): last 3 moves per player, read directly off the already-loaded `move_history` array/session reducer state — a UI slice only, no DB table, query, or extra persistence beyond what item 12 already stores.
18. **Live-game visibility** (your addition, several related pieces; updated for multiple concurrent games per room):
    - Room card badge (same spot as item 6's presence-count tweak) shows a count/list when a room has one or more active chess games — **since multiple pairs can each have a game running (item 12), this needs to be a small picker/list, not a single boolean** — styled differently per row for "you're a player" vs. "just spectate." Opening the room surfaces that picker so a third player can choose which game to watch.
    - A persistent badge on the main ribbon/collapsed tab (same affordance as SideDock's existing "claimable rewards" badge on the closed tab) shows the player has an ongoing game, visible even while they're elsewhere in the app or out of that room. (This one stays a simple boolean — it's scoped to games *you're* in, and item 10/12 already cap that at one active game per opponent, though you could still have concurrent games against different opponents; if so, badge just needs a count too.)
    - **Minimize control**: the Chess panel collapses to a small floating chip instead of closing — the game/subscription stays alive in the background so it doesn't block other desktop work; reopening restores the board. Minimizing never forfeits.
    - Leaving the ROOM (disconnecting from the realtime channel) only pauses your visible presence — no forfeit, no clock to worry about since it's untimed; the game just waits for you (and per item 10, only an explicit Give-up/mutual-cancel/unreachable-fallback ever resolves it — never a silent disconnect). Leaving the GROUP (removing membership) is bigger — since the game is stored against `group_id`, warn the player first if it would strand an active game.
19. **"Poke" button** (your addition) inside the Chess panel: sends a targeted notification to the opponent via the existing [useNotifications.ts](apps/desktop/src/online/useNotifications.ts) user-inbox system (new `NotificationKind: "chess_poke"`, which also needs adding to the hardcoded kind-whitelist check in that file), surfacing as the same toast-on-top-of-the-pet UI already used for friend requests/room invites ([GameView.tsx](apps/desktop/src/game/GameView.tsx) ~L2116-2147) — reaches them even if they've minimized the board or left the room, as long as their app is open (same accepted "missed if fully offline" tradeoff the notification system already documents). **Deep-link on tap (your call):** the poke payload carries `groupId` + the chess game's id; tapping the toast switches the viewer into that room (joining its realtime channel if they're not already in it) and restores the minimized Chess panel for that specific game — reusing item 18's minimize-restore state rather than a fresh open. If switching rooms would leave the viewer's *current* room, treat it the same as any other room switch elsewhere in the app (no special forfeit concern since leaving a room never forfeits, per item 18).
20. New `apps/desktop/src/game/minigames/Chess.tsx` bringing 14-19 together: CSS-grid board (no new art beyond the pet-as-king reuse), legal-move highlighting via chess.js, resign/give-up button (Phase 1's hook), read-only spectator mode for non-participants, resumes from DB on room (re)join.
21. Challenge flow extends the existing `challenge()`/`inviteMinigame()` targeted-broadcast pattern in `useRoom.ts` (~L762-845) — no mode picker needed now that it's untimed-only.

**Decisions**
- No physics/rendering library added — framer-motion only, per your call.
- Chess ships **untimed-only** for v1 — the clock/time-bank mode is dropped (was "both from day one", revised per your follow-up).
- Chess trust model: client-validated + DB turn-ownership RLS only for MVP; full server-side re-validation deferred as hardening follow-up.
- Forfeit excludes Battle by your choice.
- `poopCleanedCount` syncs to the cloud pets row from day one (achievements/stats will depend on it later) — not local-only.
- Poop cleanup is only possible post-hatch; eggs can't poop.
- The old daily normal-point cap is confirmed dead code (kept only for save-schema compatibility) — no interaction with poop's point trickle to worry about beyond the existing evolution-pacing check.
- Idle liveliness (breathing + future gestures) moves fully to framer-motion for better long-term scaling, with an explicit reset requirement to avoid clobbering the existing CSS-class transform states.
- Give-up always produces a decisive win/loss; a silent app-close never does. Chess adds a separate mutual-cancel (and unreachable-opponent fallback) flow that ends a game with no score impact for either side.
- Chess allows multiple concurrent games per room (different pairs), but only one active game per pair at a time.
- `chess_games` spectator RLS must use `is_group_member(group_id)`, not `shares_group_with(...)` (the latter takes a user id, not a group id).
- `chess.js` license is BSD-2-Clause (corrected from an earlier "MIT" mislabel).
- Pet death mid-chess-game is an accepted unhandled edge case for v1.
- `useRibbonPrefs` stays local-only/per-machine — not synced to the cloud.
- Poke toast taps deep-link into the correct room and restore the minimized board, rather than being a passive reminder.
